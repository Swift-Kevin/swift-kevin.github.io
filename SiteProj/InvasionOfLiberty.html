Invasion Of Liberty

Contributions

Utilizing the Vulkan API. Managing HLSL Vertex/Fragment Shaders. Debugging Tools. Developer Lighting Toggle. Game Pause State with Enemy/Bullet responsiveness.
Created and implemented assets. Source Control Management with Github. Clarified and Refactored Code during Code Freezes. Managing multiple viewports for information (lives/minimap/score).

Team Role: Graphics Programmer / Secondary Lead

Team Size: 4

Development Time: 1 month

Using Vulkan
  • Chose Vulkan due to my experience with it from the previous class.
    •  
  • We reused my previous class's renderer which helped speed up production and testing.
    •  
  • Benefitted from having clarity in code structure from my previous renderer with debugging options.
Learning/Using Shaders
  • Using Vertex and Fragment shaders to manipulate data sent to the GPU instead of CPU based unparallel computations.
    •  
  • By having the shaders already setup with descriptor sets, passing any extra needed data could be easily added. This is how I added in 3 additional viewports and toggling lighting in the fragment shader.
    •  
  • Due to time constraints I implemented the multiple viewports pragmatically. Prioritized quick and effective solutions over elaborate, time-consuming ones.
Debugging Tools
  • Adding in tools like Wireframe mode or Lighting Toggle helped denote problems related to the shaders/rendering pipeline/model materials.
    •  
  • Cycling through colors in wireframe helped discern if objects were being obfuscated or improperly drawn.
    •  
  • We also had console logging for anytime an action was performed. Entity hurt, shoot, death, sounds aswell as player location and score tracking in console.
Pausing Functionality
  • Defender (1981) didn't have a way to pause the game and we wanted to correct that.
    •  
  • The pausing functionality was done, but we didn't have a way to show it to the user. We didn't even plan on adding it in since we were low on time.
    •  
  • I spent a couple hours on the last night crunching away to display the model properly above the player wherever they are because I felt like it wouldn't be right without it.
Github and Code Refactoring
  • We used Github as our main source control point and worked on separate branches to avoid conflicts.
    •  
  • I was the most proficient at resolving Merge Conflicts so I was in charge of handleing them if they ever arose.
    •  
  • When we were going to merge into the main branch we always did a merge request and had other people view the code and test it to avoid any errors or improper implementations.
Managing Multiple Viewports
  • I used a viewport for each section of the display I needed. Lives, Minimap, Score, Gameplay were in their own viewport.
    •  
  • Each viewport had their own drawing functionality since they each required something different to them. They all have their own objects needing to be drawn.
    •  
  • I would need to update the pipeline each time to account for where the viewport would start and the dimensions, which meant I didn't need to recreate every aspect of the pipeline each frame.
