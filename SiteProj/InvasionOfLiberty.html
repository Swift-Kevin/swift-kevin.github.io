Using Vulkan

The first problem we faced was what graphics API we wanted to use. Our team ended up picking Vulkan due to my familiarity with it from the previous class (3D Content Creation).

By using my renderer, our team had a head start working on the project and seeing results. My renderer was built with a couple of debugging options that I expanded upon and we used throughout this month (Lighting/Wireframe). The renderer was setup to show my thought process with checking and abstracting functionality into respective headers/classes and how Vulkan's pipeline was laid out.

Learning/Using Shaders

Before we use the pipeline, lets make sure we have what we need!

When using any graphics rendering API you are going to need Shaders, mainly Vertex and Fragment. I would end up coming back to these often when I needed to create a new viewport and not a new shader.

The decision to not create and implement the switching of shader pipelines for multiple effects (viewports, textures, etc) was my own. I felt it was going to be research and development time that we didn't have for this project due to constraints (time allotted). Though creating and using the shaders I already implemented proved useful as I could pass along an id to flip on a specific case in either shader through the push constants buffer I had.

A lot of the implementation within the Shaders are done because of time constraints that we had to work around. A quick and dirty solution that worked completely was a lot better than a neat and organized solution that was half works.

Debugging Tools

Now how are we going to make sure things are rendering properly?

With those earlier mentioned debug tools the team can test to make sure that things are working properly within the rendering pipeline. Commonly done in the development cycle, debugging shortcuts helped my team by narrowing down where issues arose, for example if the player wasn't showing up or if the projectiles didn't properly collide.

Wireframe was added to help debug polygon rendering with instancing/indexed vertcies that were being passed to the GPU. After showing the game to others (classmates and friends), they loved the idea of having it in the game and being able to see the cool design of it. Wireframe multicoloring was added in the first installment of the renderer to allow for visibility testing with colors in the renderer.

Lighting was added to discern directional and ambient light values, making sure that they were being properly calculated within the Fragment Shader without having to use RenderDoc (Graphics API Debugger) prematurely.

Pausing Functionality

Too many enemies, I need a break!

Our inspiration, Defender (1981) was an endless space shooter where you fought for a high score. We wanted the same approach but with the ability to pause the game to calm the momentum of the game.

My teammate Jerry and I worked on pausing the game, he worked on implementing the enemies, bullets, and I worked on rendering the text to the screen. The paused text is a 3D text model that renders where the player is located at and rotate towards the camera which provides a nice depth to the text.

When rendering the model, after adding all the current models to draw there is a check to see if the game is paused or not, if it is then it adds the text model to the back of the drawing pool so it's drawn ontop of everything else.

Github and Code Refactoring

Like any good project should, use source control!

The source control platform we used was Github. We picked Github because we were all familiar with the basics of Github and how to use it in a group setting thanks to our Midterm Project a couple months prior.

I was a bit more confident in my skills with Github since I had a bit more practice in my spare time dealing with merge conflicts and attempting to resolve them to my best abilities. I was the main person for accepting merge requests into the main branch for our project since I would look over all the changes and verify that they met the requirements given.

With merge requests, our group had a rule where we do not accept our own merges unless approved by the team lead. Since I was the secondary lead this meant that any of the merges the team lead would do, I would have to verify just to ensure that it was approved by a second person and my merges to them.

Managing Multiple Viewports

How would you go about displaying different sections of the game?

I decided to use multiple viewports and use a camera matrix for each one to show important information regarding the game such as current HP, the minimap of enemy positions, and the players score.

How each viewport is decided was based off of the design we were going off of. Since each section displayed should be roughly 33% of the windows width space we can use that to calculate the positions and sizes of each section.

The image attached shows with abstraction that each viewport was dealt with individually and had its own functionality tied with itself. All I needed to do was pass along the type to make sure I was using the correct dimensions and then draw all the required objects.
